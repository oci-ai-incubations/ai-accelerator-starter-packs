# Kubeconfig Patching for OCI CLI Profile

## Overview

`oci ce cluster create-kubeconfig` generates a kubeconfig file that uses the OCI CLI as a credential exec plugin. However, it does **not** include the `--profile` flag in the generated exec args. This means kubectl will always use the `DEFAULT` OCI CLI profile for authentication, regardless of which profile you used to generate the kubeconfig.

**Freedom level: LOW — add the profile args in exactly the right location.**

---

## Symptom

After generating a kubeconfig and running `kubectl get pods`, you see:

```
error: You must be logged in to the server (the server has asked for the client to provide credentials)
```

This happens even when:
- The cluster OCID is correct
- The OCI CLI works fine with `--profile SANJOSE` (or other non-default profile)
- The kubeconfig file exists and has the correct cluster endpoint

The root cause is that `oci ce cluster create-kubeconfig` emits an exec block without `--profile`, so the OCI CLI credential helper uses `DEFAULT` profile credentials, which may not have access to the cluster.

---

## The Fix

Add `--profile` and the profile name into the exec `args` list in the kubeconfig YAML, immediately before the `env: []` line.

### Before (generated by OCI CLI)

```yaml
users:
- name: user-<cluster-id>
  user:
    exec:
      apiVersion: client.authentication.k8s.io/v1beta1
      args:
      - ce
      - cluster
      - generate-token
      - --cluster-id
      - ocid1.cluster.oc1.us-sanjose-1.aaaa...
      - --region
      - us-sanjose-1
      command: oci
      env: []
```

### After (manually patched)

```yaml
users:
- name: user-<cluster-id>
  user:
    exec:
      apiVersion: client.authentication.k8s.io/v1beta1
      args:
      - ce
      - cluster
      - generate-token
      - --cluster-id
      - ocid1.cluster.oc1.us-sanjose-1.aaaa...
      - --region
      - us-sanjose-1
      - --profile
      - SANJOSE
      command: oci
      env: []
```

Replace `SANJOSE` with the actual OCI CLI profile name for the target tenancy.

---

## Full Generate + Patch + Verify Sequence

### Step 1 — Generate the kubeconfig

```bash
export OCI_CLI_PROFILE=SANJOSE

oci ce cluster create-kubeconfig \
  --cluster-id <cluster-ocid> \
  --file ~/.kube/config \
  --region us-sanjose-1 \
  --token-version 2.0.0 \
  --kube-endpoint PUBLIC_ENDPOINT \
  --profile SANJOSE
```

Note: `--profile SANJOSE` controls which profile is used to make the `create-kubeconfig` API call, but it does NOT embed `--profile` into the generated exec args.

### Step 2 — Patch the kubeconfig

Use `sed` to insert `--profile` and the profile name before `env: []`:

```bash
PROFILE_NAME="SANJOSE"
KUBECONFIG_FILE="$HOME/.kube/config"

# Insert --profile and the profile name before "      env: []"
sed -i '' "s/      env: \[\]/      - --profile\n      - ${PROFILE_NAME}\n      env: []/" "${KUBECONFIG_FILE}"
```

Or patch manually by opening the file and inserting the two lines as shown in the "After" example above.

### Step 3 — Verify connectivity

```bash
kubectl get nodes
# Expected: list of OKE worker nodes with status Ready

kubectl get pods -A
# Expected: list of pods across all namespaces
```

If you still see credential errors after patching, verify:
1. The profile name in the kubeconfig matches exactly what's in `~/.oci/config`
2. The OCI CLI profile has the correct private key and fingerprint configured
3. The cluster's security list or network security group allows inbound traffic on the API endpoint port (6443)

---

## Notes

- This patch is required for every new kubeconfig generation — OCI CLI does not persist the fix.
- If multiple clusters are configured in the same kubeconfig, each `users` entry for OKE clusters needs the patch.
- The `--kube-endpoint PUBLIC_ENDPOINT` flag is required; without it, OCI CLI defaults to the private endpoint which is unreachable from developer machines.
